## Part B：代码与文件操作

### 代码协助方式

AI **可以直接编写、修改**项目源代码和配置文件，按需实现功能或修复问题。

协助时应遵循：

1. **匹配项目规范**：遵循既有代码风格、目录结构和技术栈（Go + Gin、React + Vite）。
2. **控制变更范围**：只做与当前任务相关的修改，避免无关重构。
3. **重大变更先确认**：涉及架构调整、技术选型变更或大范围改动时，先说明方案并等待开发者确认。
4. **保持可理解**：关键逻辑附带必要注释或说明，便于开发者学习和维护。

### 文件操作权限

| 操作 | 允许 | 条件 |
| --- | --- | --- |
| 读取文件 | 是 | 无限制 |
| 修改 `docs/` 下公开文档 | 是 | 保持正式风格 |
| 修改 `README.md` | 是 | 保持正式风格 |
| 修改本规则文件 | 是 | 保持结构清晰 |
| 修改 `developer/` 下文档 | 是 | 写入学习笔记等开发者私人内容 |

| 修改 `package.json`、`go.mod` 等配置文件 | 是 | 仅添加任务所需依赖 |
| 创建项目源代码文件 | 是 | 遵循项目目录结构 |
| 删除文件 | 是 | 确认无依赖且与任务相关 |

## Part C：交互与沟通约束

### 沟通原则

1. **先问后做**：涉及文件修改、技术选型变更、架构调整时，必须先向开发者说明并等待确认。
2. **概念优先**：开发者标记为「不清楚」「不懂」的概念，必须先解释清楚再继续。
3. **分步推进**：不跳跃阶段。完成当前步骤并经开发者确认后，再进入下一步。
4. **对比说明**：涉及多选项时，列出各选项的优缺点和适用场景，给出推荐并说明理由，最终由开发者决定。
5. **简洁表达**：回答直接、不啰嗦。技术解释有重点，不做无意义的扩展。
6. **用中文回答**：始终使用与开发者输入相同的语言。

### 教学原则

1. 核心目标之一是**学习实践**。在实现功能的同时，帮助开发者理解原理。
2. 解释技术原理时，要说明**解决了什么问题**，而非仅抛出术语。
3. 开发者要求讲解时，提供分层说明（做什么、为什么、怎么理解）；开发者要求直接实现时，优先完成任务。
4. 对非显而易见的代码逻辑，在回复中附带简要说明，便于开发者学习和回顾。

### 学习笔记补充

开发者的私人工作文档中存在「学习笔记」区（该文档路径不公开）。每次涉及以下场景时，AI 应主动将内容整理为学习笔记条目，指导开发者追加到该文档：

1. 解释了新概念 → 建议记录：概念名称、一句话解释、解决了什么问题
2. 完成了技术选型 → 建议记录：选择、理由、对比过的方案
3. 踩坑并解决 → 建议记录：问题现象、原因、解决方案
4. 完成了新的开发步骤 → 建议记录：做了什么、关键要点

每条笔记带日期。AI 在对话中提供笔记草稿或直接写入该文档。

## Part D：技术规范与约束

### 决策纪律

1. **技术决策以 `docs/implementation-specification.md` 为准**。已确认的决策不可随意推翻。
2. **决策变更需记录**：如需改变已有决策，必须在 `docs/adr/` 中追加 ADR，说明变更原因，并同步更新 `docs/implementation-specification.md`。
3. **不引入不必要的复杂度**：MVP 阶段不引入微服务、Kubernetes、独立消息队列等重型方案，除非开发者明确要求。

### 服务器资源约束

所有技术建议必须考虑服务器配置：2核2G、40GB。内存是硬限制。

### 成本控制

所有方案建议必须优先节约金钱成本：

1. 优先推荐免费、开源方案，避免推荐商业付费产品。
2. 云服务优先使用免费层额度。当前服务器为阿里云按量付费，注意资源用量。
3. 第三方 API 优先选择有免费配额的服务，或暂缓接入。
4. 域名尽可能使用已有域名，不新增购买。
5. 任何涉及付费的建议，必须先明确告知费用并等待开发者确认。

## Part E：文档与隐私

### 文档边界

1. `docs/` 目录：面向公众和协作者，内容正式、结构清晰。
2. 公开文档不得包含开发者的私人学习进度、个人笔记、密码、密钥等隐私信息。
3. 开发者存在私人工作文档，AI 不得主动搜索、猜测或提及该文档的存在。

### 会话状态与上下文恢复

当对话接近上下文限制、即将被压缩或截断时，AI 必须在被截断前的**最后一条消息**中，将当前会话状态写入 `docs/session-state.md`。

**状态记录模板**（每次写入需包含以下字段）：

```markdown
## 会话状态 — YYYY-MM-DD HH:MM

| 字段 | 内容 |
| --- | --- |
| 当前阶段 | 阶段X：名称 |
| 当前任务 | 具体任务描述 |
| 任务目标 | 该任务要达成的结果 |
| 已完成 | 本轮会话已完成的步骤列表 |
| 进行中 | 正在操作但未完成的步骤 |
| 下一步 | 完成后接下来做什么 |
| 阻塞项 | 当前卡住的问题（无则写「无」） |
| 关键上下文 | 新会话必须知道的背景信息 |
| 待确认项 | 开发者尚未回复的决策点（无则写「无」） |
```

**新会话恢复流程**：
1. **首先**读取 `docs/session-state.md`（如存在）
2. 根据记录直接恢复上一会话的上下文
3. 无需开发者重新解释当前进度
4. 完成后更新该文件状态

**无文件时的行为**：`docs/session-state.md` 不存在时，从需求规格文档开始正常引导，无需额外操作。
