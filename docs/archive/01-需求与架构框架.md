# Oxelia51 需求与架构框架

## 1. 项目愿景

该平台 Oxelia51 将成为一个统一的在线服务中心，用于整合你的软件工具、人工智能能力和 GitHub 仓库管理能力，同时作为开发者的个人作品集展示。平台需要服务两类用户：

| 用户群体 | 目标 |
| --- | --- |
| 在线访问者 | 发现并使用选定工具，这些工具可以作为公开 Web 服务或需认证的 Web 服务提供。 |
| 开发者/管理员 | 管理工具、AI 服务、GitHub 仓库、服务器资源、用户、权限、监控、部署以及未来商业化能力。 |

平台应展示技术能力，支持持续学习，提供有价值的服务，并保留向付费功能演进的路径。

## 5. 用户角色

| 角色 | 描述 | 主要权限 |
| --- | --- | --- |
| 访问者 | 浏览网站的匿名用户 | 查看公共页面，并在匿名限制内使用公共工具。 |

第一阶段中，`管理员` 与 `开发者` 可以是同一个人，但授权模型仍应按可分离原则设计。

## 6. 用户流程

### 6.1 访问者工具使用流程

1. 访问者打开平台落地页。
2. 访问者浏览工具目录。
3. 访问者选择一个工具。
4. 平台检查是否允许匿名使用。
5. 访问者提交输入。
6. 平台校验输入，创建工具任务，并以同步或异步方式执行。
7. 访问者获得结果、状态、下载链接、跳转地址或报告。
8. 平台记录使用指标并应用配额规则。

### 6.2 开发者工具接入流程

1. 开发者扫描本地 `code` 目录并识别一个项目。
2. 开发者将项目关联到 GitHub 仓库。
3. 开发者创建工具注册记录。
4. 开发者定义服务类型、路由、运行时需求、密钥、存储和权限。
5. 开发者配置公开可见性、功能开关和版本。
6. 平台执行健康检查和部署检查。
7. 开发者将工具发布到工具目录。

### 6.3 AI 任务流程

1. 用户或管理员选择一个 AI 辅助操作。
2. 平台解析权限和上下文边界。
3. AI 路由器选择提供商、模型、提示词配置和参数。
4. 按策略过滤或阻断敏感数据。
5. 按超时、重试和降级规则执行模型请求。
6. 校验并存储结果。
7. 用户审阅结果，特权操作必须显式审批。
8. 平台记录使用量、成本、延迟和审计事件。

### 6.4 GitHub 仓库管理流程

1. 管理员连接 GitHub OAuth 或 GitHub App。
2. 平台同步仓库、分支、提交、Issue、PR、Release 和工作流。
3. 管理员将仓库映射到工具记录。
4. 平台跟踪仓库健康状态、最新版本、CI 状态和部署状态。
5. 管理员触发扫描、同步任务、AI 评审或发布流程。

### 6.5 服务器部署流程

1. 管理员配置域名、DNS、TLS、反向代理、服务器凭据和部署目标。
2. 平台或 CI/CD 构建应用制品。
3. 部署先推送到预发布环境。
4. 健康检查验证就绪状态。
5. 管理员批准生产环境部署。
6. 监控系统跟踪资源使用、错误、延迟和回滚信号。

## 7. 推荐系统架构

### 7.1 初始架构

第一个可运行版本建议采用模块化单体加后台 Worker 的架构。该方式可以降低复杂度，同时保留未来扩展能力。

| 层级 | 推荐组件 | 职责 |
| --- | --- | --- |
| 前端 | Vue 或 React | 公共页面、工具 UI、管理仪表盘。 |
| 后端 API | Java Spring Boot、Go 或 FastAPI | 认证、工具注册、AI 编排、GitHub 同步、管理 API。 |
| 数据库 | PostgreSQL | 用户、工具、仓库、任务、日志、设置、审计记录。 |
| 缓存/队列 | Redis | 限流、会话缓存、后台任务协调。 |
| 存储 | 本地存储优先，后续接入 MinIO/S3 | 上传文件、工具输出、日志、制品。 |
| 反向代理 | Nginx、Caddy 或 Traefik | HTTPS 终止、路由、静态文件、请求限制。 |
| CI/CD | GitHub Actions | 构建、测试、扫描、发布、部署。 |
| 监控 | Prometheus/Grafana/Loki 或托管等价方案 | 指标、日志、告警、仪表盘。 |

### 7.2 未来架构

当流量和复杂度增加时，可拆分为以下服务：

| 服务 | 职责 |
| --- | --- |
| 身份服务 | 用户、会话、OAuth、角色、权限。 |
| 工具注册服务 | 工具元数据、状态、版本、功能开关、运行时配置。 |
| 工具执行服务 | 任务调度、执行、结果存储、配额。 |
| AI 服务 | 提供商抽象、路由、提示词、策略、用量监控。 |
| GitHub 集成服务 | OAuth/App 安装、Webhook、仓库同步、发布跟踪。 |
| 服务器运维服务 | 部署目标、健康检查、指标、资源策略。 |
| 审计服务 | 不可变日志、安全事件、可导出的治理记录。 |

## 8. 工具注册模型

`code` 目录下的每个项目最终都应转化为规范化的工具记录。

| 字段 | 含义 |
| --- | --- |
| 工具 ID | 稳定的内部标识符。 |
| 名称 | 公开展示名称。 |
| Slug | URL 安全的服务路径。 |
| 描述 | 面向访问者的说明。 |
| 仓库 | GitHub owner/name 和分支。 |
| 本地路径 | `code` 目录下的本地开发路径。 |
| 分类 | 文件、生产力、自动化、AI、开发者工具、监控等。 |
| 可见性 | 公开、需要登录、仅白名单、仅管理员。 |
| 状态 | 草稿、测试中、已启用、已禁用、已废弃。 |
| 版本 | 语义化版本或发布标签。 |
| 运行时 | 静态、API 服务、Worker 任务、容器、外部服务。 |
| 功能 | 已启用功能列表和功能开关。 |
| 配额 | 匿名/用户/管理员限制。 |
| 依赖 | 数据库、Redis、对象存储、外部 API、AI 提供商。 |
| 健康检查 | 用于验证服务可用性的 URL 或命令。 |
| 安全策略 | 输入限制、文件限制、访问规则、审计要求。 |

## 14. 安全框架

| 风险 | 控制措施 |
| --- | --- |
| API 密钥泄露 | 密钥管理器、加密、脱敏、轮换、不暴露给客户端。 |
| 恶意文件上传 | 类型限制、大小限制、沙箱执行、杀毒集成、过期清理。 |
| 未授权管理访问 | MFA、RBAC、IP 限制或 VPN、审计日志。 |
| GitHub Token 滥用 | GitHub App、最小权限、限定安装范围、Webhook 签名校验。 |
| 提示词注入 | 上下文过滤、工具权限边界、特权操作人工审批。 |
| 工具滥用 | 限流、配额、黑名单、异常检测、滥用报告。 |
| 部署失误 | 预发布环境、健康检查、回滚、受保护分支。 |
| 数据丢失 | 定时备份、恢复测试、对象生命周期策略。 |

## 16. MVP 建议

### 16.1 MVP 范围

第一个版本应避免过度建设。推荐 MVP 如下：

| 能力 | 是否纳入 MVP |
| --- | --- |
| 项目/工具注册表 | 是 |
| 手动扫描本地 `code` 目录 | 是 |
| GitHub 仓库映射 | 是 |
| 公共工具目录 | 是 |
| 管理仪表盘 | 是 |
| 用户角色 | 基础管理员/用户/访问者角色 |
| AI 提供商注册表 | 基础提供商/模型配置，不在源代码中存储真实密钥 |

| 商业化 | 仅做数据模型准备 |

### 16.2 建议首个工具

**DormGuard**（原 `dorm-power-guard-lite-repo`）技术最成熟，但按既定决策**仅作作品集案例展示**，不通过 API 网关提供在线服务。

首个接入的**在线工具**从以下项目中按优先级选定（均运行于 Oxelia51 统一 UI + API 网关）：

| 候选 | 说明 |
| --- | --- |
| SuperRead | RSS 抓取 + AI 简报，验证定时任务与用户 API Key 模式 |
| AIHelper | 提示词处理，验证模板管理与 LLM 调用 |

## 18. 技术决策点

实现前必须确认以下决策：

| 主题 | 待决策事项 |
| --- | --- |
| 后端语言 | Java Spring Boot、Go、Python FastAPI 或 Node.js/NestJS。 |
| 前端框架 | Vue 或 React。 |
| 数据库 | 推荐 PostgreSQL；如偏好 MySQL 需确认。 |
| 部署方式 | 优先 Docker Compose，或直接进程部署。 |

| 用户账号模型 | 先仅管理员使用，或 MVP 起支持公开注册。 |
| 商业化时机 | 仅做准备，或早期即实现支付。 |
