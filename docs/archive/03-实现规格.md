### 2.2 工具运行模式（内嵌 vs 独立服务 vs 容器 vs 外部 API）

选择「工具在主应用里怎么跑」，有以下四种方式：

| 模式 | 含义 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 内嵌模块 | 工具代码直接写在主项目里，作为主应用的一部分运行 | 最简单，一个进程全搞定 | 工具崩溃可能拖垮主应用 |
| 独立服务 | 工具是一个单独的进程，通过 HTTP/gRPC 和主应用通信 | 隔离性好，互不影响 | 需要管理多个进程 |
| 容器（Docker） | 每个工具打包成 Docker 容器，独立运行 | 环境隔离、部署方便 | 学习 Docker 有一定曲线 |
| 外部 API | 工具本身就是一个第三方服务，主应用只做转发 | 不需要自己维护 | 依赖外部服务 |

**对你项目的影响**：
- **DormGuard** 本身已有后端 + 前端，它是一个**独立服务**（但按决策仅作作品集案例，不做在线接入）。
- 你未来的工具（文件转换、短链接等）也各自是独立服务。

**建议**：采用「独立服务」模式，每个工具保持自己的代码库和启动方式。平台主应用通过 URL 路由把请求转发到对应工具。后续统一用 Docker Compose 编排。

### 2.7 DNS 管理是什么

**问题**：你的域名 DNS 在哪里管理？

**简单解释**：DNS 就是把域名翻译成 IP 地址的服务。你买了域名后，需要在一个地方设置「oxelia51.com 指向 47.108.202.199」。
- 如果域名在阿里云买的，DNS 大概率在阿里云。
- 你也可以把 DNS 托管到 Cloudflare（免费，且自带 CDN 和安全防护）。

**你需要做的事**：登录你买域名的平台，找到 DNS 设置页面，确认你的域名管理在哪里。记录下来。

**你的情况**：DNS 目前在阿里云，你想知道能否迁移到 Cloudflare。

**回答**：可以，而且推荐。步骤如下：
1. 注册 Cloudflare 账号（免费）
2. 在 Cloudflare 添加你的域名（如 `oxelia51.com`）
3. Cloudflare 会扫描现有 DNS 记录并自动导入
4. Cloudflare 会给你两个新的 Nameserver 地址
5. 回到阿里云域名控制台，把 Nameserver 改为 Cloudflare 提供的地址
6. 等待 DNS 生效（通常几分钟到几小时）

**好处**：免费 CDN、免费 SSL 证书自动续期、DDoS 防护、访问速度更快。你的域名在阿里云买的，但 DNS 解析可以交给 Cloudflare，两者不冲突。

### 2.10 Redis、对象存储、Docker

- **Redis**：一个内存数据库，主要做三件事——缓存（加快访问速度）、消息队列（异步处理任务）、限流（防止滥用）。你的 MVP 需要它来做「会话管理」和「限流」。建议引入。
- **对象存储**：存文件的地方（类似网盘）。你的工具涉及文件上传时才会用到。MVP 阶段用服务器本地磁盘即可，**暂不引入**。
- **Docker**：把应用打包成「集装箱」，到哪里都能跑。帮你解决「我电脑上能跑，服务器上不行」的问题。建议 MVP 就引入。
可以

| GraphQL | 前端想要什么数据就查询什么，只有一个入口 | 更适合复杂查询场景 |

**建议**：**REST 优先**。

## 七、架构决策补充

以下为 2026-07-04 新增的关键架构决策：

### 7.1 工具调用模式：API 网关（ADR-002）

**决策**：平台统一承担所有工具的 UI 渲染，工具仅提供后端 API。

| 层级 | 职责 |
|------|------|
| 平台前端（React） | 渲染所有页面：落地页、工具目录、每个工具的操作界面 |
| 平台后端（Go） | 用户认证、工具注册、API 网关：接收前端请求→转发工具后端→返回结果 |
| 各工具 | 仅提供后端 API，无独立前端 |

用户始终在 `oxelia51.com` 下操作，无跳转、无分离感。各工具 API 仅内网可达。

### 7.2 仓库可见性（ADR-003）

| 仓库 | 可见性 | 写权限 |
|------|--------|--------|
| `Oxelia51`（平台） | 公开 | 仅拥有者及授权协作者 |
| 各工具项目 | 公开 | 各仓库独立管理 |

敏感信息不提交仓库，通过 `.env` 和环境变量管理。

### 7.3 项目更名

`developer-tool-service-platform` → `Oxelia51`（2026-07-04）
